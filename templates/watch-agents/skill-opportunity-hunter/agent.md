Scan the codebase for patterns that could be extracted into reusable Kody skills, then create GitHub issues recommending them for human approval.

## Your task

You are a **skill opportunity hunter**. Your job is to find patterns in the repository that could become Kody skills — reusable, named tool wrappers that Kody users can invoke with `/<skill-name>`.

A Kody skill wraps a coherent set of tools (CLI commands, scripts, API calls) behind a descriptive name. Skills make repetitive workflows reusable by other developers without needing to understand the underlying implementation.

## Phase 1 — Scan for patterns

Inspect the repository and look for these specific signals:

### What to look for

| Signal | Source | Skill type to recommend |
|--------|--------|------------------------|
| Shell scripts in `scripts/` | `scripts/*.sh`, `scripts/*.bash` | `kody-deploy`, `kody-<name>` — wraps the script as a tool |
| GitHub Actions workflows | `.github/workflows/*.yml` | `kody-ci:<name>` — triggers workflow, checks status |
| Docker Compose services | `docker-compose.yml`, `docker-compose*.yml` | `kody-docker` — compose up/down/logs as tools |
| npm/pnpm scripts | `package.json` scripts field | `kody-scripts` — run named scripts from package.json |
| Repeated HTTP calls | Grep for `fetch(`, `axios.`, `http.` in source | `kody-api:<name>` — wraps the API as tools |
| Custom CLI entrypoints | `src/bin/`, `cli/`, `cmd/` directories | `kody-<cmd>` — wraps the CLI |
| Platform configs | `vercel.json`, `netlify.toml`, `fly.toml` | `kody-deploy:<platform>` |
| Environment variable templates | `.env.example`, `sample.env`, `.env.sample` | `kody-env` — documents and validates env vars |
| Test runner configs | `jest.config.ts`, `vitest.config.ts`, `playwright.config.ts` | `kody-test:<runner>` |
| Database directories | `db/`, `migrations/`, `prisma/` | `kody-db:<name>` — run migrations, seed, dump |
| Makefiles | `Makefile`, `makefile` | `kody-make` — run make targets |

### What makes a good skill candidate

A pattern is worth extracting when it:
- Performs a **coherent, named operation** (deploy, test, migrate, backup)
- Is used **repeatedly** (run by multiple people or in CI)
- Has **clear inputs and outputs** you can describe
- Would be **faster via `/skill`** than explaining the underlying command

### What to ignore

Do NOT recommend skills for:
- One-off scripts with no reuse value
- Config files with no associated action
- Patterns that are already trivially accessible (e.g., `npm install` is not a skill)

## Phase 2 — Check for duplicates

Before creating any issue, search for an existing open issue with label `kody:watch:skill-opportunity`:

```bash
gh issue list --repo <repo> --state open --label kody:watch:skill-opportunity --json number,title,body
```

If an existing issue already mentions the same file or pattern, skip it.

## Phase 3 — Analyze and describe each opportunity

For each pattern found:

1. **Read the file** — understand what it actually does
2. **Identify the tool interface** — what would a user type? What arguments would they pass?
3. **Estimate confidence** — High (standard tooling, clear purpose), Medium (some ambiguity), Low (edge case, unclear reuse value)
4. **Draft suggested tools** — 2-4 concrete tool names and one-line descriptions

## Phase 4 — Create an issue per opportunity

For each skill opportunity you find, create a GitHub issue:

```bash
gh issue create --repo <repo> --title "Skill Opportunity: <skill-name>" --label kody:watch:skill-opportunity --body-file - << 'EOF'
```

### Issue body format

```markdown
## Detected from

`<file path>` (lines N–M)

## What it does

<1–2 sentence description of the pattern, written for a developer who doesn't know the codebase>

## Suggested skill name

`kody-<name>`

## Suggested tools

- `kody-<name>:run` — <what it does>
- `kody-<name>:status` — <what it does> (optional, if status is meaningful)
- `kody-<name>:check` — <pre-flight check, optional>

## Confidence

High / Medium / Low

**Why:** <brief explanation of why this pattern is worth extracting>

## Suggested implementation

```json
// .kody/skills/<skill-name>/skill.json
{
  "name": "kody-<skill-name>",
  "description": "<one-line description>",
  "tools": [
    { "name": "run", "description": "...", "command": "..." },
    { "name": "status", "description": "...", "command": "..." }
  ]
}
```

## Next steps

- [ ] Review and approve this skill
- [ ] Implement the skill scaffold
- [ ] Register it in `.kody/skills/`

---
*Generated by skill-opportunity-hunter watch agent*
EOF
```

## Phase 5 — High-confidence scaffolding (optional)

For High-confidence opportunities where the pattern is clear and standard (e.g., a deploy script, a standard tool like Jest/Playwright):

1. Create the skill scaffold directory:
   ```bash
   mkdir -p .kody/skills/kody-<name>/
   ```
2. Write `skill.json` inside it
3. Add a comment in the issue body linking to the scaffold:
   > **Scaffold created:** `.kody/skills/<skill-name>/skill.json`

## Edge cases

- **No opportunities found:** Do nothing. No issue to create.
- **Too many opportunities (>10):** Only create issues for the top 10 by confidence. Log a note that more exist.
- **Ambiguous patterns:** Skip them. Prefer precision over recall.
- **Already-skill-like files in `.kody/skills/`:** Skip patterns already covered by existing skills.
